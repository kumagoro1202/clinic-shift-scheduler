## 1. エグゼクティブサマリー

本調査は、**日勤のみのクリニック**（外来診療所）向けシフトスケジューリングシステムの改善・再設計を目的として実施した。OSSライブラリ・アルゴリズム手法・商用SaaS製品・ドメイン要件・学術文献の5領域を並列調査した結果、以下の重要な知見が得られた。

現行のPuLP（MILP）+ CBCソルバー実装は**日勤のみのクリニックとして適切な選択**であり、基本アーキテクチャは維持できる。最も即効性の高い改善は**CBCからHiGHSソルバーへの切り替え**（1行変更・2〜5倍高速化）である。看護課シフトを追加する際も、**日勤のみ前提であれば夜勤変数・夜勤インターバル制約は完全に不要**となり、変数数が夜勤ありと比べて大幅に少なく抑えられる。長期的には**Google OR-Tools CP-SAT**への移行が拡張性・性能の観点で最も推奨される。競合SaaS製品と比較して、現実装はアルゴリズム的優位性（最適解の数学的保証）を持つ一方、スタッフ向けUIと希望収集機能が不足している。

## 2. 現状のPuLPベース実装の評価（日勤前提での再評価）

### 2.1 強み

| 評価項目 | 内容 |
|---------|------|
| **アルゴリズム適合性** | LP/MILPは日勤のみ制約（シフト種別が少ない・変数が少ない）に対して最適解を数学的に保証できる |
| **既存資産の活用** | コードベースが既に稼働しており、移行コストゼロで継続利用できる |
| **ライセンス** | MITライセンスで商用利用自由 |
| **スケール対応** | 日勤のみ・20〜50名・1ヶ月分なら現行CBCでも数秒〜数十秒で解ける |
| **制約追加の柔軟性** | 新しい制約を線形式で追加するだけ。クリニック固有の要件に直接対応可能 |

### 2.2 弱み

| 課題 | 内容 |
|------|------|
| **ソルバー性能** | デフォルトのCBCソルバーはHiGHSより2〜5倍遅い。1行変更で即時改善可能 |
| **軟制約の優先度付け** | 「できれば○○したい」という希望の重み付けが煩雑。OR-Tools CP-SATの方が自然に記述できる |
| **スタッフUIの欠如** | 希望休の収集・シフト確認をスタッフが行う手段がなく、管理者の手入力負荷が高い |
| **スケールの天井** | 100名超・複雑な多部門制約では限界が現れ始める（日勤のみであれば当面問題なし） |

### 2.3 看護課シフト追加時の課題（日勤のみ前提で）

**日勤のみであるため、看護課追加のハードルは大幅に低い**。主な課題は以下の3点に絞られる：

1. **スキル制約の拡張**: 看護師・准看護師・助産師など資格別の配置制約を線形式で追加する。医師1名につき1〜2名の看護師配置制約が核心
2. **診察補助・処置担当の時間帯分離**: 医療事務の受付対応時間帯と、看護師の診察補助時間帯の同時確保を制約化する
3. **昼休憩カバーのローテーション**: 昼休憩中も受付を絶やさないためのカバー制約が必要

**除外できる要素（日勤のみのため完全不要）**:
- 夜勤インターバル制約（夜勤終業〜次勤務11時間以上）
- 夜勤後翌日休み保証
- 連続夜勤禁止
- 深夜割増賃金の計算
- 夜勤専従スタッフの特別管理
- 2交代/3交代の切り替え管理

**結論**: 現行PuLPアーキテクチャでも看護課シフトは実装可能。**CBCをHiGHSに切り替えた上で**スキル制約・配置制約を拡張することを推奨する。

## 3. 推奨アルゴリズム・ライブラリ（TOP3）

### 🥇 第1位: Google OR-Tools（CP-SAT）— 移行推奨

| 項目 | 内容 |
|------|------|
| **言語** | Python / Java / C# / C++ |
| **ライセンス** | Apache-2.0（商用利用自由） |
| **GitHub** | 13.1k★（2026年1月 v9.15最新） |
| **日勤クリニック適合度** | ◎◎◎ |

**選定理由**:
1. **公式ナーススケジューリングサンプルあり**: `shift_scheduling_sat.py`（看護師スケジューリングの完全実装例）を公式提供。日勤への適用はさらにシンプル
2. **業界最高水準の性能**: 日勤クリニック（20〜50名・1ヶ月分）なら数秒以内に最適解を算出
3. **軟制約の記述が自然**: 希望休などの重み付けを`model.maximize()`で統一的に扱える
4. **将来拡張性**: スタッフ数増加・複数診療科追加にも安定して対応
5. **`pip install ortools`のみ**: インストールが簡単

**移行コスト**: 中程度（変数定義・制約の記述方法が変わるため、段階的移行を推奨）

```python
# OR-Tools CP-SAT シフト変数定義例（日勤クリニック向け）
from ortools.sat.python import cp_model
model = cp_model.CpModel()
# shifts[(staff, day, shift_type)] = 1 → そのスタッフがその日にそのシフトに入る
shifts = {}
for s in range(num_staff):
    for d in range(num_days):
        for t in shift_types:  # 例: ['full', 'am', 'pm', 'off']
            shifts[(s, d, t)] = model.new_bool_var(f'shift_s{s}_d{d}_t{t}')
```

### 🥈 第2位: PuLP（現行維持）+ HiGHSソルバー — 即効性最高

| 項目 | 内容 |
|------|------|
| **言語** | Python |
| **ライセンス** | MIT |
| **移行コスト** | 最低（1行変更のみ） |
| **日勤クリニック適合度** | ◎◎ |

**選定理由**:
1. **既存コードを維持**: リファクタリング不要で即日適用可能
2. **ソルバー切り替えのみで高速化**: CBCからHiGHSに変えるだけで2〜5倍の速度改善
3. **日勤のみ規模では十分**: 看護課追加後も1ヶ月分の計算が数秒〜数十秒以内に完了する見通し

**改善実装（1行変更）**:
```python
# 変更前（CBC）
prob.solve(PULP_CBC_CMD())

# 変更後（HiGHS — 2〜5倍高速）
from pulp import HiGHS_CMD
prob.solve(HiGHS_CMD())
```

## 4. 競合SaaSとの機能比較

### 4.1 日勤クリニック向け製品との比較

| 機能 | shift-scheduler-claude | らくしふ | シフオプ | SHIFTEE |
|------|----------------------|--------|--------|---------|
| **AI自動シフト生成** | ◎（ILP最適解保証） | △（限定的） | ○（AIアシスト） | ○（オプション） |
| **制約のカスタマイズ** | ◎（コードで自由設計） | ✕ | △（UI設定のみ） | △ |
| **スタッフ希望収集UI** | ✕ | ◎（LINE連携） | ◎（スマホアプリ） | ○ |
| **モバイル対応** | ✕ | ◎ | ◎ | ◎ |
| **スキル管理** | ○（コードで設定） | △ | △ | ○ |
| **有給消化管理** | △ | ○ | ○ | ○ |
| **診療カレンダー連動** | ◎（自由設計） | △ | △ | △ |
| **夜勤特化機能** | 不要（日勤のみ） | △ | △ | △ |
| **コスト** | 運用コストのみ | 要見積 | 300円/名/月 | 200〜500円/名/月 |

### 4.2 現在の実装で優れている点

1. **最適解の数学的保証**: 商用SaaSの多くは「AIアシスト型」であり最適解を保証しない。PuLPはMILP最適解を保証する
2. **制約の完全カスタマイズ**: クリニック固有のルール（診療科別最低配置・特定スキルの専従ルール等）をコードで自由に定義可能
3. **診療カレンダー完全連動**: 臨時休診・院長学会出張日等の不規則な休診日を自由に設定できる
4. **コスト**: SaaSの月額費用が発生しない。プライバシー上のデータ外部送信もない

### 4.3 不足している主要機能

1. **スタッフ向け希望収集UI**: LINEやスマホアプリによる希望申請機能がない（管理者が手入力）
2. **シフト通知・共有機能**: スタッフへのシフト通知・確認機能がない
3. **リアルタイム変更対応**: 当日の急な欠勤に対してリスケジューリングする仕組みがない
4. **有給消化の自動提案**: 有給残日数を参照して自動的に休暇を提案する機能がない
5. **勤怠管理連携**: 打刻・残業管理との連携がない

## 5. 日勤クリニック重要要件（まとめ）

### 5.1 現行実装に追加すべき制約

**ハード制約（HC）— 違反するとシフト不成立**:

| ID | 制約内容 | 優先度 |
|----|---------|--------|
| HC-001 | 診療時間帯に医師1名以上の在院確認（医療法・診療所開設要件） | 最高 |
| HC-002 | 診療時間帯に看護師（または准看護師）1名以上の配置 | 最高 |
| HC-003 | 診療時間帯に医療事務1名以上の受付配置 | 高 |
| HC-004 | 連続勤務6日以内（労基法・4週4日休日確保から逆算） | 高 |
| HC-005 | 週労働時間40時間以内（または変形労働時間制での月単位管理） | 高 |
| HC-006 | 法定休憩の確保（6時間超→45分、8時間超→60分） | 高 |
| HC-007 | スキル要件の遵守（採血・処置等は看護師免許保有者のみ） | 高 |

**ソフト制約（SC）— 違反するとペナルティが加算**:

| ID | 制約内容 | 推奨重み |
|----|---------|---------|
| SC-001 | スタッフの希望休充足（月○日分） | 高（100） |
| SC-002 | 土曜出勤の均等分散（差 ≤ 1回/月） | 高（100） |
| SC-003 | 月間勤務日数の標準偏差最小化（σ ≤ 1.0日） | 中（50） |
| SC-004 | 不人気シフト（遅番等）の均等配分 | 中（50） |
| SC-005 | 4日以上連続勤務の最小化 | 中（30） |
| SC-006 | 有給消化促進（年5日取得義務への対応） | 中（30） |
| SC-007 | 新人と指導員の同時配置 | 低（10） |

### 5.2 看護課実装で特に注意すべき点

1. **医師:看護師の比率管理**: 医師1名につき1〜2名の看護師が目安。診療科・時間帯別の最低配置数を制約として設定する
2. **准看護師の扱い**: 准看護師は看護師の指示のもとでのみ業務可能。単独で処置担当にするとハード制約違反になる
3. **特定行為看護師**: 高度な医療行為には特定行為研修修了者が必要。スタッフにskillフラグを追加して管理する
4. **昼休憩のローテーション**: 診療昼休憩中も受付カバーが必要な場合、看護師と医療事務でローテーション制約を設計する
5. **夜勤なし＝設計の単純化**: 夜勤インターバル・夜勤明け翌日休み・深夜割増は完全に除外できる。これにより変数数は夜勤ありと比較して大幅に少なく抑えられる

## 6. 再設計時の推奨アプローチ

### 6.1 段階的移行 vs フルスクラッチの比較

| 観点 | 段階的移行（推奨） | フルスクラッチ |
|------|-----------------|--------------|
| **リスク** | 低（既存動作を維持しながら改善） | 高（全機能を再実装） |
| **実装期間** | 短〜中（フェーズ別に実施） | 長（設計・実装・テスト全やり直し） |
| **ソルバー性能** | Phase1でHiGHS切り替え（即効性高） | 最初からOR-Tools CP-SAT採用可能 |
| **ROI** | 高（小さな変更で大きな改善） | 低（投資対効果が読みにくい） |
| **推奨度** | ★★★★★ | ★★☆☆☆ |

**→ 段階的移行を強く推奨する。**

### 6.2 優先度付きロードマップ案

```
Phase 1（即時・低リスク）
  ├─ [1日以内] CBCソルバーをHiGHSに切り替え（1行変更・2〜5倍高速化）
  ├─ [1ヶ月以内] 看護課シフトの追加（スキル制約・配置制約の拡張）
  └─ [1ヶ月以内] 有給消化促進ロジックの追加（年5日義務への対応）

Phase 2（中期・スタッフUX改善）
  ├─ スタッフ希望収集フォーム（Web/LINE連携）
  ├─ シフト結果のスタッフ向け通知機能
  └─ 月間公平性レポート（スタッフ別充足率・土曜出勤回数）

Phase 3（長期・アーキテクチャ進化）
  ├─ OR-Tools CP-SATへの段階的移行（軟制約の記述改善・スケール対応）
  ├─ LLMによる自然言語制約入力UI
  │    例: 「田中さんは今週水曜午後を休みに」→ 自動制約変換
  └─ リアルタイム再スケジューリング機能（急な欠勤への当日対応）
```

### 6.3 評価指標（KPI）

| 指標 | 目標値 |
|------|--------|
| ハード制約充足率 | 100%（1件でも違反 → シフト不成立） |
| スタッフ希望充足率 | 80%以上 |
| 月間勤務日数の標準偏差 | σ ≤ 1.0日 |
| 土曜出勤回数の差分 | 差 ≤ 1回/月 |
| シフト生成時間（1ヶ月分） | 30秒以内（HiGHS切り替え後） |
| 有給取得率 | 年5日以上（法的義務） |
