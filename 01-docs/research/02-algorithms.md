## 各手法詳細

---

### 1. 制約充足問題（CSP: Constraint Satisfaction Problem）

#### 概要・仕組み

シフト割り当てを「変数・ドメイン・制約」の3要素で定式化する。

```
変数   X = { x_{i,d} | 職員 i の 日 d の勤務種別 }
ドメイン D = { 早出, 通常, 遅出, 休暇, 午前休, 午後休 }
制約   C = { ハード制約（必須）, ソフト制約（望ましい） }
```

探索アルゴリズムとして **バックトラッキング**（違反時に前の変数へ戻る）や
**弧整合性（AC-3）**（ドメインを事前に削減して探索空間を縮小）を使用。
Google OR-Tools の CP-SAT ソルバーは SAT 技術と組み合わせた高性能実装。

#### 強み・弱み

| 強み | 弱み |
|------|------|
| ハード制約の表現が直感的 | ソフト制約の「重み付き最適化」が ILP より不得意 |
| 実行可能解の存在チェックが容易 | 大規模問題では探索空間が爆発 |
| OR-Tools CP-SAT は高速・実績豊富 | 純粋なCSPでは最適性の保証が難しい |

#### 日勤クリニックへの適合性: ◎

- 夜勤なしのため制約数が少なく、変数・ドメインが小さい → **高速に解ける**
- 「この職員は水曜午後は必ず休み」「エリアAに最低2名」などのハード制約表現が得意
- 職員20〜50名・1ヶ月なら CP-SAT で秒単位で解ける

#### 実装難易度: 中

#### 代表的なライブラリ

- **Google OR-Tools CP-SAT**（Python/Java/C++）— 最も実績豊富
- **python-constraint** — 軽量、小規模向け
- **Choco Solver**（Java）— 企業向け実績多数

### 2. 整数計画法（ILP/MIP: Integer Linear Programming）

#### 概要・仕組み

シフト割り当てを **バイナリ変数** x_{i,d,s} ∈ {0,1} で表し、
線形の目的関数と制約式で最適化問題を定式化する。

```
変数   x_{i,d,s} = 1 → 職員 i が 日 d に シフト s を担当

目的関数（例: スキルスコアを最大化）
  maximize: Σ skill(i, s) × x_{i,d,s}

制約（例）
  ∀d, s: Σ_i x_{i,d,s} ≥ required(d, s)   # 必要人数を確保
  ∀i, d: Σ_s x_{i,d,s} ≤ 1                 # 1人1日1シフト
  ∀i, d ∈ vacation(i): Σ_s x_{i,d,s} = 0  # 休暇日は勤務不可
```

#### ソルバー比較

| ソルバー | ライセンス | 性能 | 日勤クリニック向け評価 |
|---------|-----------|------|----------------------|
| **HiGHS** | MIT（完全無料） | ★★★★ | **◎ 最推奨**（高速・OSS） |
| **CBC (COIN-OR)** | EPL（無料） | ★★★ | ◎ 現状実装（十分） |
| **GLPK** | GPL（無料） | ★★ | ○ 小規模なら可 |
| Gurobi | 商用（学術無料） | ★★★★★ | △ ライセンス管理が煩雑 |
| CPLEX | 商用 | ★★★★★ | △ 同上 |

**日勤クリニック（20〜50名）では HiGHS か CBC で十分**。
商用ソルバーの性能差は大規模問題（500変数超）で顕著になるため、この規模では不要。

#### 強み・弱み

| 強み | 弱み |
|------|------|
| **最適解の数学的保証** | 変数数増加で計算時間が指数的に増加（NP困難） |
| 制約を線形式で正確に表現できる | 非線形ソフト制約の表現が難しい場合がある |
| PuLP/OR-Tools など豊富なラッパー | 大規模（100名超）では限界あり |
| ソルバーの改善が速い（特にHiGHS） | 定式化に数学的知識が必要 |

#### 日勤クリニックへの適合性: ◎

- 夜勤なし = 変数数が少ない（2交代制の半分以下）
- 日勤クリニックは「シフト種別数」が少ない（早出/通常/遅出/休暇 程度）
- 職員30名・1ヶ月（約900変数）: HiGHS/CBC で **数秒以内**に最適解が出る
- **現在の shift-scheduler-claude（PuLP+CBC）はこの手法を採用しており、適切な選択**

#### 実装難易度: 中

```python
# PuLP + HiGHS での実装例
from pulp import LpProblem, LpVariable, LpMaximize, lpSum, HiGHS_CMD

prob = LpProblem("DaytimeClinicShift", LpMaximize)
x = LpVariable.dicts("shift",
    [(i, d, s) for i in employees for d in days for s in shifts],
    cat='Binary')

# スキルスコアの最大化
prob += lpSum(skill[i][s] * x[i, d, s]
             for i in employees for d in days for s in shifts)

# 必要人数制約
for d in days:
    for s in shifts:
        prob += lpSum(x[i, d, s] for i in employees) >= required[d][s]

prob.solve(HiGHS_CMD())  # CBC より 2〜5倍高速
```

#### 代表的なライブラリ

- **PuLP**（Python）+ HiGHS/CBC — shift-scheduler-claude 採用
- **OR-Tools MIP**（Python/Java）
- **scipy.optimize.milp**（Python 3.9+）— 軽量

### 3. ルールベース・貪欲法（Rule-Based / Greedy）

#### 概要・仕組み

最適化エンジンを使わず、**ルールとヒューリスティクスで順次割り当て**る。

```
貪欲法の例:
  for 日付 d in 1..31:
      for エリア a in 必要エリア:
          空き職員を優先度順（スキルスコア降順）でソート
          上位から必要人数分を割り当て
          休暇・すでに割り当て済みの職員はスキップ
```

**ルールベースの例**:
- 「スキルの高い職員を月曜に優先配置」
- 「先週末出勤した職員は今週末を休みに」
- 「連続5日以上勤務した場合は翌日を休みに」

#### 強み・弱み

| 強み | 弱み |
|------|------|
| 実装が最もシンプル | **最適解の保証なし**（局所最適に陥りやすい） |
| 高速（ミリ秒単位） | ルールの組み合わせで矛盾が生じやすい |
| 動作が説明しやすい | 複雑な制約には対応困難 |
| 最初のプロトタイプとして有効 | ルール追加のたびに他のルールとの整合性チェックが必要 |

#### 日勤クリニックへの適合性: ○（小規模・シンプルな制約）

- 職員5〜10名・制約が少ないなら実用的
- 制約が複雑になるほど ILP/CSP の方が優れる
- **初期実装や緊急時の代替手段として価値がある**
- shift-scheduler-claude の現規模（20〜30名）では ILP に劣る

#### 実装難易度: ★★★★★（最易）

### 4. メタヒューリスティクス

最適解の保証はないが、大規模問題でも高速に良質な近似解を得られる手法群。
日勤クリニック規模（20〜50名）では ILP/CSP で十分なケースが多く、**補完的な位置づけ**。

---

#### 4.1 遺伝的アルゴリズム（GA: Genetic Algorithm）

**仕組み**: 複数のシフト案（個体）を「選択→交叉→突然変異」で進化させる。

```
染色体 = [シフト種別] × (職員数 × 日数)
例: [早, 休, 通, 遅, 早, 通, ...] — 職員全員の31日分を1列に並べた配列

適応度 = スキルマッチスコア
       - 制約違反ペナルティ × 重み
```

| 強み | 弱み |
|------|------|
| 局所最適を抜けやすい（多様性維持） | パラメータチューニングが必要 |
| 並列化が容易 | 最適解の保証なし |
| 複雑なソフト制約も適応度関数で対応可 | 収束判定が難しい |

**日勤クリニック適合性: ○**
夜勤なしのため制約が単純 → ILP で十分。GAは職員数100名超の大規模化後に有効。

---

#### 4.2 焼きなまし法（Simulated Annealing: SA）

**仕組み**: 温度 T を徐々に下げながら解を探索。一定確率で悪化方向への移動も許容し、局所最適を脱出。

```python
T = T_initial
while T > T_min:
    s' = 近傍操作(s)  # 例: 2人のシフトを交換
    Δ = fitness(s') - fitness(s)
    if Δ > 0 or random() < exp(Δ / T):
        s = s'         # 改善なら必ず採用、悪化でも確率的に採用
    T *= cooling_rate  # 温度を徐々に下げる
```

| 強み | 弱み |
|------|------|
| 実装がシンプル | 冷却スケジュールの設計が重要 |
| ILPの初期解生成に使える | 最適解の保証なし |
| 日勤クリニック規模なら高速 | パラメータ感度が高い |

**日勤クリニック適合性: ○**

---

#### 4.3 タブーサーチ（Tabu Search）

**仕組み**: 直近の移動履歴をタブーリストに記録し、逆戻りを禁止しながら系統的に探索。

```
1. 近傍 N(s) を全列挙
2. タブーリストにない最良の近傍 s' を選択
3. s' をタブーリストに追加（一定期間後削除）
4. 収束まで繰り返す
```

**日勤クリニック適合性: ○**
SA より系統的。ただし日勤クリニック規模では ILP の方が実装・性能ともに優れる。

### 5. AI/機械学習ベースアプローチ

---

#### 5.1 強化学習（RL: Reinforcement Learning）

**仕組み**: シフト割り当てをエージェントの逐次的な意思決定問題として定式化。

```
状態:   現在のシフト割り当ての部分解
行動:   次の1つの割り当て（職員 × 日 × シフト）を決定
報酬:   制約充足度・スキルマッチ度・公平性のスコア
```

採用アルゴリズム: DQN・PPO・A3C 等

| 強み | 弱み |
|------|------|
| 繰り返しパターンの学習が得意 | 学習コスト（時間・計算資源）が大きい |
| 一度学習すれば推論が速い | 制約違反のハンドリングが困難（ペナルティで近似） |
| 複雑なソフト制約に対応可能 | ハイパーパラメータ調整が難しい |

**日勤クリニック適合性: △（過剰）**

- 日勤クリニック規模では ILP で十分。RL 導入のコストに見合わない
- 職員数が100名超・毎日リアルタイムでシフト修正が必要、といった場合に検討価値あり
- 研究段階。実務導入事例はまだ少ない

---

#### 5.2 LLMを活用したアプローチ（2025-2026 最新動向）

**主な活用パターン**:

**パターン1: 自然言語 → 制約変換（補助ツールとして有望）**
```
管理者入力: 「田中さんは今週水曜の午後を休みにしてください」
    ↓ LLM が解釈・構造化
→ ILP の制約コードに変換 → ソルバーで最適化
```

**パターン2: LLM が直接シフトを出力（実験的）**
```
プロンプト: 「以下の職員・制約に基づきシフトを生成してください...」
→ LLM が直接シフト表を出力（小規模・単純制約では実用的）
```

**パターン3: 説明生成・UI補助**
```
最適化結果 → LLM が「田中さんは月・水・金に早出です。スキルスコアが最高のため...」
と自然言語で説明を生成
```

| 強み | 弱み |
|------|------|
| 自然言語インターフェース | 制約充足の保証なし |
| 実装が柔軟 | ハルシネーション（制約の捏造）リスク |
| 非技術者の管理者でも使いやすい | API コスト・プライバシーリスク |

**日勤クリニック適合性: △（補助的）**

- 単体では最適化ツールに及ばない
- ILP/CSP と組み合わせた補助 UI ツールとして価値がある
- 将来的に「自然言語でシフト制約を入力する UI」として統合が有望

---

#### 5.3 Graph Neural Network（GNN）

**仕組み**: シフトスケジューリングをグラフ問題として定式化し、GNN が割り当て確率を学習。

```
ノード: 職員・日付・シフト種別
エッジ: 「この職員はこの日にこのシフトを担当できるか」の関係
GNN → ノードの特徴量を集約・学習 → 割り当て確率を出力
```

**日勤クリニック適合性: ×（過剰）**

- 研究段階。実務導入事例がほぼない
- 大量の学習データが必要（小規模クリニックでは非現実的）
- 日勤クリニックの規模・複雑さに対して完全に過剰スペック

## 日勤クリニックに最適な手法ランキング（根拠付き）

### 🥇 第1位: ILP/MIP（整数計画法）— PuLP + HiGHS/CBC

**根拠**:
1. **最適解の数学的保証** — 職員のスキル配置・公平性を最大化できる
2. **日勤のみの制約は少ない** — 変数数が少なく、20〜50名・1ヶ月なら秒単位で解ける
3. **制約の追加が容易** — 新しい要件（看護課対応・部署分離等）を線形式で追加するだけ
4. **shift-scheduler-claude が既に採用** — PuLP + CBC は動作実績あり。HiGHS に切り替えで2〜5倍高速化可能
5. **OSS で無料** — HiGHS（MIT）、CBC（EPL）はライセンスフリー

**具体的な改善提案**:
```python
# 現状（CBC）
prob.solve(PULP_CBC_CMD())

# 推奨（HiGHS — より高速）
from pulp import HiGHS_CMD
prob.solve(HiGHS_CMD())
```

---

### 🥈 第2位: CSP（制約充足問題）— OR-Tools CP-SAT

**根拠**:
1. OR-Tools CP-SAT は ILP に匹敵する性能を持つ
2. 「実行可能解があるか」の判定が ILP より速い
3. ソフト制約の重み付け最適化も可能（maximize/minimize）
4. Google がクリニック・病院向けシフトの公式サンプルを提供

ILP とほぼ同等。どちらを選ぶかは既存コードの継続性による（現状 ILP なので ILP を継続推奨）。

---

### 🥉 第3位: ルールベース・貪欲法

**根拠**:
1. **プロトタイプ・緊急時の代替手段** として価値がある
2. 職員5〜10名・制約が少ない段階では十分実用的
3. 実装が最もシンプル — 医療現場のスタッフでも理解しやすい
4. ILP が失敗した場合のフォールバックとして組み込む価値あり

**制限**: 複雑な制約（看護課+事務課の同時最適化等）には対応困難。

---

### ❌ 採用非推奨（日勤クリニック規模では過剰）

| 手法 | 非推奨理由 |
|------|-----------|
| GA・SA・タブーサーチ | ILP で最適解が出る規模。メタヒューリスティクスの複雑さは不要 |
| 強化学習 | 学習コスト・実装複雑さが見合わない。研究段階 |
| GNN | 学習データ不足・研究段階。完全に過剰 |
| LLM単体 | 最適性保証なし。補助 UI として使うなら価値あり |

---

## shift-scheduler-claude への具体的推奨

1. **短期**: HiGHS ソルバーへの切り替え（`pip install highspy`、PuLP 対応）
2. **看護課実装時**: 変数が2倍になるが HiGHS/CBC で対応可能。LNS（大規模近傍探索）は不要
3. **将来 UI 改善**: LLM を「自然言語制約入力」の補助として ILP と組み合わせる
4. **スケール上限**: 職員100名超になった時点で LNS またはCSP/ILP のハイブリッドを検討
