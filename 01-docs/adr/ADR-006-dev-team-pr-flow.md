# ADR-006: 開発体制・PR承認フローの確立

| 項目 | 内容 |
|------|------|
| **ステータス** | 承認済み |
| **日付** | 2026-03-01 |
| **意思決定者** | 殿（方針指示）、軍師（設計）、3ペルソナ（SP1/SP2/SP3）（レビュー） |

## コンテキスト

シフト作成システムv2は、殿（エンジニア）が将軍システム（org-shogun）を通じて開発する。
副院長は要求事項の提供者であり、技術的なPRレビュー・承認は行わない。
この体制を明文化し、PR承認フローを確立する必要がある。

## 決定事項

1. **開発指揮系統**: 副院長（要求提供）→ 殿（最終意思決定）→ 将軍システム（開発実行）→ リポジトリ
2. **PR最終承認者**: 殿のみ。副院長はPRを見ない
3. **自動マージ例外**: ADR-004で定義した軽微な変更（typo、lint修正、パッチ依存更新）は殿の承認なしで自動マージ可能
4. **CLAUDE.md見落とし防止**: 4層構造で足軽のルール見落としを構造的に防止

### PR承認フロー

| ステップ | 担当 | 内容 |
|---------|------|------|
| 1. PR作成 | 足軽 | 実装完了後にPR作成（AIクロスレビュー結果付き） |
| 2. 機械チェック | CI | markdownlint, テスト, 静的解析 |
| 3. 最終承認 | **殿** | PR内容確認 → Approve → マージ |

### CLAUDE.md見落とし防止（4層構造）

| 層 | 手段 | 内容 |
|----|------|------|
| 1 | context/{project}.md | org-shogunのコンテキストにルール要約を記載 |
| 2 | タスクYAML必須フィールド | `claude_md_read: MANDATORY` + `pre_work_checklist` |
| 3 | CLAUDE.md集約 | このファイルにルール・ワークフロー・禁止事項を集約 |
| 4 | cmd発行時自動適用 | 家老がプロジェクトを検知し、自動でルールをタスクYAMLに付与 |

## 代替案と却下理由

| 代替案 | 却下理由 |
|--------|---------|
| 副院長もPRレビューに参加 | 副院長はエンジニアではない。技術的なPRレビューは負担であり価値が低い |
| 足軽が直接マージ | 品質管理のゲートがなくなる |
| CLAUDE.md防止策なし（足軽の自己責任） | 構造的にCLAUDE.mdの読み忘れが発生する |

## 根拠

1. PR承認フローの明確化により、責任の所在が不曖昧になる
2. 4層の防止策により、単一の見落としが問題に発展しない
