# ADR-007: 個人情報保護多層防御の採用

| 項目 | 内容 |
|------|------|
| **ステータス** | 承認済み |
| **日付** | 2026-03-01 |
| **意思決定者** | プロジェクトオーナー（方針指示）、設計レビューエージェント（設計）、3ペルソナ（SP1/SP2/SP3）（レビュー） |
| **重要度** | 【MANDATORY】 |

## コンテキスト

シフト作成システムv2はGitHubパブリックリポジトリとして公開される。
副院長からの要求事項にはスタッフ実名等の個人情報が含まれる可能性がある。
パブリックリポジトリに個人情報がコミットされた場合、git履歴に残り、削除が極めて困難。

## 決定事項

**3層の多層防御アーキテクチャ**を採用:

| Layer | 防壁 | 検出タイミング | ブロック方式 |
|-------|------|--------------|------------|
| 1 | 匿名化ルール（CLAUDE.md + 教育） | コード記述時 | AIエージェントがルールに従い匿名表記を使用 |
| 2 | Pre-commit Hook（ローカル） | コミット前 | パターンマッチで検出 → コミットブロック |
| 3 | CI/GitHub Actions（リモート） | Push/PR時 | パターンマッチで検出 → CIをFAIL → マージブロック |

付帯決定:

- **姓ブロックリスト**: 日本の姓トップ100を `scripts/names-blocklist.txt` で管理
- **パターン一元管理**: `scripts/pii-patterns.txt` に正規表現を集約
- **allowlist管理**: 変更はPR必須。直接mainコミット禁止
- **commit-msgフック**: コミットメッセージ内の個人情報もチェック対象
- **インシデント対応計画**: `01-docs/spec/PII_INCIDENT_RESPONSE.md` に事故時の対応手順を文書化
- **リポジトリ初日から有効**: 最初のコミットからpre-commit hook と CI を含める

## 代替案と却下理由

| 代替案 | 却下理由 |
|--------|---------|
| Layer 1のみ（ルール教育のみ） | 人間もAIもルールを忘れる。自動検出なしでは漏洩リスクが高い |
| Layer 2のみ（pre-commit hookのみ） | `--no-verify` でバイパス可能。CIの最終防壁がないと不十分 |
| プライベートリポジトリにする | パブリック公開が前提。個人情報保護とリポジトリ公開は両立させる |

## 結果と影響

- 個人情報漏洩のリスクが構造的に低減
- 開発者（AIエージェント）がpre-commit hookに引っかかる場面が増えるが、これは意図した動作
- CI実行時間がわずかに増加（パターンスキャン追加分、数秒程度）

## 根拠

1. パブリックリポジトリでの個人情報漏洩は**取り返しがつかない**（git履歴に残る）
2. 多層防御により、単一の防壁の失敗が即座に漏洩につながらない
3. False Positive許容方針: 個人情報漏洩の被害はFalse Positiveの不便を圧倒的に上回る
4. パターン一元管理により、メンテナンスコストを抑制
