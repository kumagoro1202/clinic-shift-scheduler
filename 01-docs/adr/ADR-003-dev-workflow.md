# ADR-003: 開発ワークフローの設計方針

| 項目 | 内容 |
|------|------|
| **ステータス** | 承認済み |
| **日付** | 2026-03-01（初版） → 2026-03-01（方針変更により再設計） |
| **変更理由** | 殿の方針変更: 現行v1の21ステップワークフローは検討段階で実運用されておらず、そのまま移植するな |

## コンテキスト

現行v1には21ステップの開発ワークフロー（22エージェント定義）が存在するが、**実運用されていない**（殿の証言 2026-03-01）。
初版ADR-003では「全面移植」を決定していたが、殿の方針変更により撤回。
以下の設計方針に基づきゼロベースで再設計する:

- 人間の確認を最小限にしつつ確実に品質担保
- AIエージェント間のクロスレビューで品質を自律担保
- 人間の介入は「要件の確認」と「最終成果物の承認」のみが理想
- 問題検出時はAIが自動で修正→再検証のサイクルを回す

## 決定事項

**ゼロベース再設計: 7ステップAI自律開発ワークフロー** を採用。（詳細はADR-004参照）

| 項目 | 内容 |
|------|------|
| ステップ数 | 7（v1の21から大幅削減） |
| エージェント数 | 5（v1の22から大幅削減） |
| 人間介入 | 2箇所のみ（要件確認・最終承認） |
| 品質担保 | 統合自動チェック + AIクロスレビュー（3ペルソナ以上） |
| 障害対応 | AI自動修正ループ（最大3回リトライ、超過時は人間エスカレーション） |

## 代替案と却下理由

| 代替案 | 却下理由 |
|--------|---------|
| v1の21ステップを全面移植 | **却下**（初版の決定を撤回）。未運用のワークフローを移植しても同じく使われない可能性が高い |
| v1を部分移植（ステップ数を減らす） | v1の構造的問題（過剰分割・3人間ゲート）は部分修正では解決しにくい |
| ワークフローなし（フリーフォーム開発） | 品質担保が個人に依存し、再現性がない |

## 根拠

1. v1の21ステップは検討段階の産物で実運用実績がない
2. 22エージェント × 21ステップの保守コストが高く、実際に使われなかった要因と推測
3. 7ステップに統合することで、`/workflow auto` で要件確認後〜最終承認前まで自律実行可能
