# ADR-005: AIクロスレビューの運用基準

| 項目 | 内容 |
|------|------|
| **ステータス** | 承認済み |
| **日付** | 2026-03-01 |
| **意思決定者** | 設計レビューエージェント（設計）、3ペルソナ（WP1/WP2/WP3）（レビューで合意） |

## コンテキスト

Step 5のAIクロスレビューにおいて、False Positive（過剰な差戻し）のリスクと、レビューの信頼性のバランスを取る必要がある。

## 決定事項

1. AIレビュー判定は **YAML形式の構造化出力** を必須とし、判定理由を透明化する
2. Critical判定のみ差戻し対象。Major以下は自動修正→再検証で進行
3. 初期運用では「寛容モード」（Critical以外は自動修正）で開始し、品質問題が発生した場合に段階的に厳格化
4. False Positive率をレポートで追跡し、高い場合はレビュー基準の緩和をADRで記録

### YAML構造化出力フォーマット

```yaml
cross_review:
  reviewer: "コード品質レビュアー"
  verdict: pass  # pass | minor_issues | major_issues | critical
  issues:
    - severity: minor
      location: "02-src/backend/solver.py:45"
      description: "変数名が不明瞭"
      auto_fixable: true
    - severity: major
      location: "02-src/backend/api.py:120"
      description: "入力バリデーションが不足"
      auto_fixable: true
  summary: "軽微な命名問題1件、中程度のバリデーション不足1件"
```

### ペルソナ構成（最低3名）

| ペルソナ | 観点 | チェック項目 |
|---------|------|-------------|
| **コード品質レビュアー** | 可読性・保守性 | 命名規則、関数サイズ、DRY原則、エラーハンドリング、テストカバレッジ |
| **セキュリティレビュアー** | 脆弱性・安全性 | インジェクション攻撃、認証/認可、入力検証、機密情報の露出 |
| **ドキュメント整合性レビュアー** | 設計⇔実装一致 | API仕様と実装の一致、設計書の更新漏れ |

## 代替案と却下理由

| 代替案 | 却下理由 |
|--------|---------|
| 単一レビュアーのみ | 観点が偏る。網羅性が低い |
| Critical/Major両方で差戻し | 過剰な差戻しでワークフローが停滞する（v1の教訓） |
| レビュー結果を自由形式テキスト | 構造化されていないと自動処理ができない |

## 根拠

1. 過剰な差戻しは開発速度の低下を招き、ワークフロー自体が使われなくなる（v1の教訓）
2. 段階的厳格化により、実運用データに基づいたチューニングが可能
3. 構造化出力により、レビュー品質の改善サイクルを回せる
