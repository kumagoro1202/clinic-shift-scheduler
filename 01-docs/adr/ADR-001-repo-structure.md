# ADR-001: リポジトリ構造の設計

| 項目 | 内容 |
|------|------|
| **ステータス** | 承認済み |
| **日付** | 2026-03-01 |
| **意思決定者** | プロジェクトオーナー（最終承認）、設計レビューエージェント（提案）、4ペルソナ（レビュー） |

## コンテキスト

シフト作成システムv2を、現行v1（shift-scheduler-claude）とは完全に別のリポジトリとして新規構築する。
ディレクトリ階層、リポジトリ名、開発基盤の選定が必要。

## 決定事項

1. **ディレクトリ階層**: ナンバリングプレフィックス方式（01-docs/02-src/03-tests）を継続。ADRディレクトリ（01-docs/adr/）を新設
2. **リポジトリ名**: `clinic-shift-scheduler`（プロジェクトオーナー承認済み 2026-03-01）
3. **Gitアカウント**: `kumagoro1202`（プロジェクトオーナー裁定 2026-03-01）
4. **CI**: markdownlint + dependabotを初期導入。テスト系CIは言語決定後に追加
5. **テスト分類**: unit/integration/e2e の種別分類（言語非依存）

## 代替案と却下理由

| 代替案 | 却下理由 |
|--------|---------|
| ナンバリングなしディレクトリ（src/, docs/, tests/） | プロジェクトの方針に反する。v1との連続性を失う |
| リポジトリ名 `shift-scheduler-v2` | バージョン番号を含む命名は長期的に陳腐化する |
| フルCIを初期から構築 | 実装言語未定の状態でテスト系CIを作ると空振りになる |
| Gitアカウント `kumanoGoro` | shift-scheduler-claude（v1）専用アカウント。v2はkumagoro1202を使用 |

## 結果と影響

- リポジトリ初期構成が確定し、即座にリポジトリ作成タスクに移行可能
- 言語選定は別cmdで進めても、ディレクトリ構成が影響を受けない設計

## 根拠

1. ナンバリングプレフィックスはv1で実運用済み。v1開発者が迷わず移行できる
2. `clinic-shift-scheduler` はドメイン（clinic）を明示。バージョン番号なしで永続的に使える名前
3. 段階的CI導入により、初期のドキュメント先行フェーズで無駄なCIを避ける
